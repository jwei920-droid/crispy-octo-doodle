# AGENTS.md

本文件用于约束 Codex 在本项目中的开发行为。

## 1. 角色定位
你是本项目的实现工程师，不负责重新定义产品方向。你需要严格依据 `docs/` 目录中的文档进行开发。

## 2. 开发目标
优先完成一个 **可运行、可演示、可继续扩展** 的微信小程序 MVP。
不要擅自添加超出一期范围的功能。

## 3. 技术约束
- 使用 **原生微信小程序**
- 代码以 TypeScript 为优先
- 后端默认采用 **微信云开发**
- 优先保证结构清晰、模块清晰、便于后续替换真实 AI 能力
- 不接入支付、登录、社交、外部视频能力

## 4. 产品范围约束
### 必须支持
- 问题记录（语音 / 文字 / 图片）
- 问题列表
- 问题详情
- 三层回答展示
- 儿童插画展示
- 收藏
- 搜索
- 每周总结

### 严禁超范围实现
- 外部视频播放
- 用户体系复杂化
- 社交分享链路
- 商业化付费系统
- 排行榜、积分、成就体系

## 5. 代码原则
- 先实现主流程，再补边界状态
- 组件职责单一
- 所有接口调用统一放在 `services/`
- 所有类型统一放在 `types/`
- 页面不直接写复杂业务逻辑，优先抽服务层
- 云函数和页面之间通过明确的数据契约交互

## 6. UI 原则
- 风格温和、简洁、儿童友好
- 避免花哨动画
- 优先保证清晰可读
- 字段顺序、文案命名遵循 `docs/pages.md`

## 7. AI 相关原则
- 严格遵循 `docs/ai-rules.md`
- 生成内容要区分：小朋友版 / 故事版 / 家长提示版
- 不允许输出过于成人化、惊悚或过长内容
- 不在前端拼接复杂提示词；提示词规则集中管理

## 8. 实现顺序
1. 项目脚手架与基础配置
2. 页面与组件骨架
3. mock 数据与主流程
4. 云函数接口骨架
5. 真实数据接入
6. AI 接入
7. 错误处理与体验优化

## 9. 输出要求
每个阶段完成后：
- 说明完成了哪些文件
- 列出未完成项
- 列出待验证项
- 不要声称“已上线”或“已可商用”，除非明确满足验收文档

## 10. 遇到不明确时
优先参考顺序：
1. `docs/acceptance.md`
2. `docs/pages.md`
3. `docs/api.md`
4. `docs/prd.md`

如果仍不明确，做 **最保守、最简单、最利于后续修改** 的实现。
